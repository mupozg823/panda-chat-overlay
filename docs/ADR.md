# ADR: PandaTV Chat Overlay Revamp Decisions

## ADR-001: 순수 HTML/CSS/JS 유지

**결정**  
React/Vue/Svelte 대신 단일 HTML 파일 + 인라인 스크립트 구조를 유지한다.

**이유**

- 사용자는 브라우저에서 HTML 파일 하나만 열면 된다.
- 빌드/배포/의존성 설치 없이 공유 가능하다.
- 최종 결과물이 CSS 문자열이므로 프레임워크 도입 이점이 제한적이다.

**트레이드오프**

- 파일 비대화가 빠르게 진행된다.
- 상태와 뷰 로직 분리가 약하다.

## ADR-002: `!important` 전략 유지

**결정**  
라이브 위젯 인라인 스타일 대응을 위해 CSS 생성 결과에 `!important` 전략을 유지한다.

**이유**

- 팬더TV 위젯은 인라인 스타일과 테마 CSS를 동시에 사용한다.
- CSS만 주입 가능한 환경에서 원본 스타일 우선순위를 밀어내야 한다.

**트레이드오프**

- 유지보수성이 낮다.
- 선택자 설계가 부정확하면 부작용이 커진다.
- 일반 인라인 스타일 대응에는 유효하지만, 더 강한 우선순위 케이스까지 무조건 이긴다고 가정할 수는 없다.

## ADR-003: 공통 규칙 전략은 "호환 레이어 + 사용자 스타일" 2단 구조로 유지한다

**결정**  
실제 결과는 "테마별 호환 레이어 + 공통 사용자 스타일"의 2단 구조로 생성한다.

**이유**

- `kakaotalk`, `box`, `roundbox`, `balloon`, `neon`은 내부 구조 차이가 크다.
- 바깥 `li`만 공통 처리하는 방식으로는 내부 배경/애니메이션/쉐도우를 충분히 제거할 수 없다.

**트레이드오프**

- CSS 생성 로직이 길어진다.
- 테마 분기 관리가 필요하다.

## ADR-004: 미리보기는 라이브 DOM을 모사해야 한다

**결정**  
미리보기용 샘플 DOM은 실제 위젯 클래스명과 내부 구조를 가능한 한 그대로 반영한다.

**이유**

- 단순 샘플은 실제 결과 예측력이 낮다.
- 미리보기와 결과 차이가 크면 설정기 신뢰도가 떨어진다.

**트레이드오프**

- 미리보기 마크업이 복잡해진다.
- 설정 변경 로직도 더 세분화된다.

## ADR-005: 정보 구조는 "호환성 우선"으로 바꾼다

**결정**  
설정기 메뉴는 색상/효과보다 호환성 옵션을 먼저 보여준다.

**권장 순서**

1. 호환성
2. 말풍선
3. 아이콘 / 캡슐
4. 레이아웃 확장
5. 후원 / 알림
6. 가시성 / 모션

**이유**

- 실제 문제는 "색상 부족"보다 "테마 충돌"이다.
- 사용자는 먼저 자신의 위젯 테마에 맞는 안정화 방식을 선택해야 한다.

## ADR-006: 범용 숨김 선택자는 제거한다

**결정**  
`.flex.flex-row.flex-wrap` 같은 광범위 유틸 조합 선택자는 제거하거나 더 안전한 대상으로 치환한다.

**이유**

- Tailwind 유틸 클래스 조합은 재사용 범위가 넓다.
- 의도 외 요소를 함께 숨길 위험이 크다.

## ADR-007: 문서 우선 개편 절차를 따른다

**결정**  
개편은 문서 정합성 확보 후 코드 개편으로 진행한다.

**순서**

1. PRD 최신화
2. DOM 분석 최신화
3. ADR 정리
4. 개편 실행 계획 수립
5. 설정기 / CSS 로직 개편

**이유**

- 현재 문제는 단순 버그보다 요구사항과 실제 구현의 괴리다.
- 문서를 먼저 맞춰야 이후 구현 판단 기준이 흔들리지 않는다.

## ADR-008: 단일 HTML 설정기는 오버레이 전용 범위로 제한한다

**결정**  
단일 HTML 설정기와 프리뷰 편집기는 팬더HP에서 이미 제공하는 텍스트 색상/폰트/굵기/그림자를 기본 UI에서 숨기고, 오버레이 전용 항목을 우선 노출한다.

**이유**

- 텍스트 편집 책임이 팬더HP와 이 도구 사이에 중복되면 사용자가 혼란스럽다.
- 이 도구의 강점은 말풍선, 캡슐, 아바타, 아이콘, 레이어, 후원/알림 같은 플랫폼 외부 커스터마이징이다.

## ADR-009: 프리뷰 직접 편집은 눈에 보이는 타깃으로 진입시킨다

**결정**  
프리뷰 직접 편집은 정밀 클릭에만 의존하지 않고, 빠른 편집 도크와 아바타/아이콘 클릭 타깃 같은 눈에 보이는 보조 UI를 제공한다.

**이유**

- `messageAvatar`, `nickSlot`처럼 pseudo-element로 렌더링되는 아바타/아이콘은 클릭이 불안정할 수 있다.
- 작은 요소를 정확히 찍어야만 편집이 열리는 구조는 스트리머 도구로서 사용성이 낮다.

## ADR-010: 설정 저장은 로컬 저장만 지원한다

**결정**  
설정 저장 / 불러오기는 브라우저 localStorage와 JSON 파일 입출력만 지원하고, 서버 저장이나 계정 연동은 추가하지 않는다.

**이유**

- 단일 HTML 파일 구조와 오프라인 사용성을 유지할 수 있다.
- PRD 비목표인 백엔드 / 계정 / 서버 저장 기능을 피하면서도 실사용에 필요한 복구 기능을 제공할 수 있다.

## ADR-011: 상단 phase UI는 가이드형 대시보드로 유지한다

**결정**  
`STEP 1 레이아웃`, `STEP 2 프리셋`, `STEP 3 오버레이 조정`은 단순 탭이 아니라 현재 선택 요약과 다음 단계 CTA를 갖는 가이드형 대시보드로 유지한다.

**이유**

- 이 도구는 옵션 수가 많아 phase UI가 단순 탭이면 현재 어디까지 끝났는지 읽기 어렵다.
- 팬더HP와 역할을 나눈 뒤에는 오버레이 전용 구조 선택과 미세조정 흐름을 더 명확히 안내해야 한다.

## ADR-012: OBS Custom CSS 검증은 obs-browser load-end 주입 모델을 기준으로 해석한다

**결정**  
OBS 브라우저 소스의 Custom CSS는 `obs-browser`가 load-end 시점에 `<style>` 태그를 `<head>`에 추가하는 방식으로 이해하고 검증한다.

**이유**

- 브라우저에서 `<style>` 태그를 주입해 본 결과와 OBS 결과를 비교하는 기준이 필요하다.
- 공식 `obs-browser` 소스 코드가 실제로 이 동작을 수행한다.

**트레이드오프**

- 브라우저 실험은 좋은 근사치지만, CEF 버전 차이와 위젯 런타임 상태까지 완전히 동일하다고 단정할 수는 없다.

## ADR-013: 테마 클래스 위치는 `ul`/`li` 두 형태를 모두 지원한다

**결정**  
CSS 생성기는 `ul.{theme} > li...`와 `ul > li.{theme}...` 두 형태를 동시에 지원한다.

**이유**

- 라이브 위젯 CSS 근거상 핵심 형태는 `li.{theme}` 계열이다.
- 프로젝트 미리보기와 일부 기존 CSS는 `ul.{theme}` 계열에 의존해 왔다.
- 둘 중 하나만 고정하면 실제 결과와 미리보기 중 한쪽이 쉽게 깨진다.

**트레이드오프**

- 생성되는 CSS 길이가 늘어난다.
- 선택자 수가 많아져 관리 비용이 조금 올라간다.

## ADR-014: 추가 커스텀 CSS는 공식 탈출구로 제공한다

**결정**  
사용자가 입력한 커스텀 CSS를 최종 출력 하단에 그대로 병합하는 기능을 공식 지원한다.

**이유**

- 외부 서비스 DOM 편차와 예외 케이스를 모두 GUI 옵션으로 흡수하는 것은 비효율적이다.
- 마지막 수동 보정 경로를 공식화하면 실전 대응력이 높아진다.

**트레이드오프**

- 사용자가 강한 오버라이드를 직접 넣을 수 있어 결과가 더 예측 불가능해질 수 있다.

## ADR-015: 로컬 서버는 검증 편의를 위해 CORS 헤더를 허용한다

**결정**  
`scripts/serve.js`는 로컬 검증 시 자산 fetch를 돕기 위해 `Access-Control-Allow-Origin: *` 헤더를 반환한다.

**이유**

- 위젯 페이지에서 로컬 아이콘과 CSS를 교차 origin으로 테스트할 때 검증이 막히지 않도록 하기 위함이다.

**트레이드오프**

- 이 헤더는 검증 편의 기능이며, 제품 백엔드 기능처럼 오해될 수 있다.
- 문서에서 "최소 서버"의 의미를 함께 명시해야 한다.

## ADR-016: data URI 자산은 경량화를 우선한다

**결정**  
기본 아이콘과 대표 프리셋 자산은 CSS 총량을 크게 늘리지 않도록 경량 리소스를 우선한다.

**이유**

- OBS 사용자 지정 CSS는 긴 문자열로 저장/복사/붙여넣기된다.
- 대표 프리셋이 수백 KB 단위 CSS를 만들면 사용자 경험이 급격히 나빠진다.

**트레이드오프**

- 더 큰 아이콘 원본을 항상 그대로 실어 나르지 않으므로, 고해상도 자산은 필요 시 별도 경로로 관리해야 한다.

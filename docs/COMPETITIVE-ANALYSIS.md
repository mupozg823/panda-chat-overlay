# 경쟁 서비스 분석

> 분석일: 2026-03-13  
> 목적: 팬더TV 채팅 오버레이 개편 우선순위 정리

## 2. 현재 프로젝트에 유효한 시사점

### 시사점 A. 경쟁력은 "옵션 수"보다 "정확한 결과 예측"이다

대형 서비스들은 옵션이 많아도 결국 아래를 잘 해결한다.

- 실제 위젯 구조를 기준으로 결과가 안정적이다.
- 사용자가 현재 쓰는 레이아웃/테마와 충돌하지 않는다.
- 미리보기와 실제 결과 차이가 크지 않다.

현재 프로젝트는 옵션 수보다 `호환성 레이어`, `실위젯 검증`, `예측 가능한 출력`이 더 중요하다.

### 시사점 B. 초기 우선순위는 신규 기능보다 호환성이다

경쟁 서비스 대비 부족한 항목은 많지만, 지금 가장 시급한 것은 아래 기능이 아니다.

- Google Fonts
- 애니메이션 종류 확대
- 설정 공유
- 원격 저장

지금 더 중요한 것은:

1. 라이브 DOM과 맞는 미리보기
2. 테마별 내부 구조 리셋
3. 위험 선택자 제거
4. OBS 브라우저 소스에서 재현 가능한 검증 경로

### 시사점 C. GUI 생성기라면 "호환성 옵션"과 "탈출구"가 같이 있어야 한다

뚜봇, StreamElements, 트윕 사례를 보면 실제 사용자는 다음을 자주 만진다.

- 배지 보이기 / 숨기기
- 페이드 시간
- 메시지 수 제한
- 애니메이션 제어
- 역할/구조별 레이아웃
- 커스텀 CSS로 마지막 예외 조정

팬더TV 프로젝트에서는 이 개념을 "호환성 모드", "테마 타겟", "추가 커스텀 CSS" 조합으로 제공하는 것이 현실적이다.

## 3. 현재 프로젝트와 비교

| 항목 | 현재 프로젝트 | 개편 필요성 |
|------|:---:|:---:|
| 미리보기 DOM 정합성 | 중간 이상 | 높음 |
| 테마별 내부 구조 대응 | 중간 이상 | 높음 |
| 위험 선택자 통제 | 중간 이상 | 중간 |
| GUI 옵션 수 | 중간 이상 | 보통 |
| 호환성 중심 메뉴 | 높음 | 중간 |
| 설정 저장/불러오기 | 있음 | 낮음 |
| 추가 커스텀 CSS | 있음 | 낮음 |
| 메시지 수 제한 | 없음 | 중간 |
| 텍스트 외곽선 | 없음 | 중간 |
| 실위젯 회귀 검증 자동화 | 없음 | 높음 |

## 4. 개편 우선순위 제안

### P0. 지금 바로 유지할 축

| 기능 | 이유 |
|------|------|
| PRD / DOM / ADR 최신화 | 구현 기준선 재정립 |
| 미리보기 DOM 현실화 | 결과 예측 정확도 확보 |
| 테마별 호환 레이어 | 실제 사용성 핵심 |
| 메뉴 IA 재구성 | 스트리머 중심 사용 흐름 확보 |
| 위험 선택자 제거 | 방송 중 사고 가능성 축소 |
| 실위젯 재검증 메모 축적 | "브라우저 미리보기만 맞는 도구"가 되지 않도록 유지 |

### P1. 호환성 개편 이후 붙일 기능

| 기능 | 참조 |
|------|------|
| 메시지 수 제한 | KapChat, StreamElements |
| 텍스트 외곽선 | 트윕, StreamElements |
| 페이드 시간 세분화 | 트윕, 뚜봇 |
| 실위젯 회귀 검증 자동화 | StreamElements 개발 워크플로우 참고 |

### P2. 장기 개선

| 기능 | 참조 |
|------|------|
| Google Fonts 통합 | StreamElements |
| 입장 애니메이션 선택 | StreamElements |
| 설정 URL 공유 | 각종 오버레이 도구 |
| 원격 프리셋 저장 | 트윕, Streamlabs |

## 5. 결론

현재 프로젝트는 이제 "기능이 없는 상태"가 아니라 "핵심 기능은 갖췄지만, 실위젯 검증을 더 구조화해야 하는 상태"에 가깝다.  
경쟁 서비스와의 차별점도 결국 "팬더TV 위젯에 특화된 정확한 CSS 생성기"가 되는 데서 나온다.

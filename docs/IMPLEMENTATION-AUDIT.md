# Implementation Audit

> 기준일: 2026-03-13  
> 범위: `overlay-settings.html`, `overlay.css`, `scripts/serve.js`, 로컬 실행/저장/검증 흐름

## 1. 결론

이 저장소는 제품 로직 기준으로는 **프론트엔드 단일 도구**다.  
`scripts/serve.js`는 정적 파일을 열기 위한 최소 서버이며, 계정/DB/API를 가진 백엔드는 아니다.

최근 이슈 중 상당수는 "서버가 없어서"가 아니라 아래에서 발생했다.

- 문서와 구현의 시간차
- 실위젯 DOM 추정의 오차
- 캐시/검사 방법 때문에 생긴 오진
- `ul-theme` / `li-theme` DOM 편차

## 2. 분류

### A. 의도된 최소 서버

- [scripts/serve.js](/Users/bagjaeseog/Downloads/_방송도구/panda-chat-overlay/scripts/serve.js)
  정적 파일 서빙만 담당한다.
- 판단: `의도된 구조`
- 이유:
  PRD/ADR 기준으로 이 프로젝트는 단일 HTML 설정기 + CSS 생성기가 목표이며, 원격 저장/API는 비목표다.
- 보완:
  현재는 로컬 교차 origin 검증 편의를 위해 `Access-Control-Allow-Origin: *` 헤더를 추가로 반환한다.
  이것은 검증 보조 기능이지 백엔드 확장이 아니다.

### B. 프론트만 구현됐지만 백엔드가 필요 없는 기능

- 브라우저 저장/불러오기: `localStorage`
- JSON 파일 저장/가져오기
- 프리뷰 직접 편집
- 커스텀 아이콘의 `fetch -> blob -> base64` 변환
- 추가 커스텀 CSS 병합
- 판단: `의도된 프론트 구현`
- 이유:
  이 기능들은 브라우저만으로 완결되며, 백엔드가 개입하면 단일 파일 도구의 장점이 약해진다.

### C. 버그/불완전 구현으로 봐야 하는 항목

- phase 대시보드가 숨김 플레이스홀더로 남아 있던 상태
  - 판단: `버그 또는 미완성 리팩토링`
  - 이유: 상태 동기화용 `id`만 남고, 사용자가 봐야 할 overview/summary UI가 `display:none`으로 바뀌어 있었다.
- phase 전환/레이아웃 선택이 `onclick*=` 문자열 매칭에 의존하던 상태
  - 판단: `취약 구현`
  - 이유: 마크업 문자열 변경에 약하고, 상태 소스가 DOM 구조가 아니라 속성 텍스트였다.
- 생성 CSS가 `ul.{theme} > li...` 구조에 과도하게 의존하던 상태
  - 판단: `호환성 버그`
  - 이유: 실제 위젯의 `li.{theme}` 구조에서 스타일이 누락될 수 있었다.

### D. 오진 또는 검증 오류로 분류한 항목

- "후원/알림이 생성 CSS 때문에 숨겨진다"는 초기 진단
  - 판단: `검증 오진`
  - 이유: 실제로는 `nav`, 원본 SVG 배지, 스크롤바 계열만 의도적으로 숨김 처리됐고, 후원/알림은 CSS 때문에 사라진 것이 아니었다.
- "공주 아이콘 data URI가 여전히 구 대용량 값이다"는 초기 진단
  - 판단: `캐시/정규식 기반 검사 오류`
  - 이유: 브라우저 캐시와 검사 패턴 문제로 잘못 읽혔고, 실제 출력은 경량화된 data URI를 사용했다.

### E. dead code / 과잉 UI로 분류한 항목

- `preview-tabs` CSS
  - 판단: `dead code`
  - 이유: 대응하는 마크업이 없었다.
- 플로팅 저장/불러오기 행, `loadSettingsFile2`, `showFloatingSaveStatus`
  - 판단: `중복 UI / 제거 대상`
  - 이유: CSS 출력 영역의 저장/불러오기와 기능이 중복됐고 화면을 더 복잡하게 만들었다.

### F. 앞으로 개발해야 하는 항목

- 브라우저/기기 간 동기화
- 팀 공유용 원격 프리셋 저장
- 다중 사용자 협업
- 실제 팬더TV 테마별 정밀 리그레션 검증 자동화
- 판단: `향후 개발`
- 이유:
  이 항목들은 현재 아키텍처를 넘어서는 범위이며, 실제로는 백엔드나 테스트 인프라가 필요하다.

## 3. 이번 정리에서 반영한 방향

- phase UI를 다시 보이는 3단계 대시보드로 복원
- 현재 선택 상태와 다음 단계 CTA를 같은 데이터 소스로 묶음
- 프리셋/카테고리/레이아웃 상태 동기화 경로 정리
- `ul-theme` / `li-theme` 이중 셀렉터 지원
- 추가 커스텀 CSS 병합과 저장 경로 연결
- 기본 아이콘 자산 경량화로 대표 프리셋 CSS 크기 축소
- 로컬 검증용 CORS 헤더 추가
- 대표 프리셋의 실제 위젯 재검증 결과를 문서에 반영

## 4. 후속 기준

앞으로 새 항목을 추가할 때는 아래 기준으로 먼저 분류한다.

1. 브라우저 단독으로 끝나는가  
   끝나면 프론트 기능으로 유지한다.
2. 원격 저장/계정/동기화가 필요한가  
   필요하면 신규 백엔드 범위로 분리한다.
3. 화면에서 보이지 않는데 상태만 남아 있는가  
   버그 또는 미완성 리팩토링으로 우선 정리한다.
4. 같은 기능 버튼이 두 군데 이상 있는가  
   편의성이 아니라면 과잉 UI로 보고 줄인다.
5. 실제 위젯에 주입해 확인했는가  
   아니면 "확인된 사실"이 아니라 "가설"로 문서화한다.

# PRD: PandaTV Chat Overlay Revamp

> 버전: 2026-03-13  
> 상태: Active Revamp  
> 관련 문서: [DOM-ANALYSIS.md](DOM-ANALYSIS.md), [ADR.md](ADR.md), [IMPLEMENTATION-AUDIT.md](IMPLEMENTATION-AUDIT.md), [REVAMP-PLAN.md](REVAMP-PLAN.md)

## 1. 제품 요약

팬더TV 스트리머가 OBS 브라우저 소스에서 사용할 수 있는 채팅 오버레이 CSS와 비주얼 설정기를 제공한다.  
개편의 핵심은 기존의 "공통 스타일 생성기"를 "실제 팬더TV 채팅 위젯 DOM과 OBS Custom CSS 동작에 맞춘 호환성 중심 생성기"로 전환하는 것이다.

2026-03-13 기준 대표 `👑 공주(princess)` 프리셋을 실제 위젯 페이지에 재주입해 후원/알림 표시, 닉네임 캡슐, 아이콘 data URI 로드를 재검증했다.

## 2. 문제 정의

현재 프로젝트는 기본적인 말풍선/텍스트/후원/알림 스타일 생성은 가능하지만 다음 문제가 남아 있다.

1. 실제 팬더TV 위젯의 테마별 내부 DOM 차이를 완전히 흡수하지 못한다.
2. 위젯 DOM에서 테마 클래스가 `ul` 또는 `li`에 붙는 편차를 안정적으로 대응해야 한다.
3. 미리보기 DOM이 라이브 위젯 DOM과 크게 어긋나면 결과 예측 정확도가 낮아진다.
4. 일부 테마는 내부 `message__text`, `heart__text`, `notice__text`에 직접 배경/그림자를 걸어 공통 바깥 래퍼 오버라이드만으로는 부족하다.
5. 프리뷰 직접 편집에서 아바타/아이콘처럼 작은 표적은 정밀 클릭이 필요해 사용성이 떨어질 수 있다.
6. 팬더HP 채팅 페이지와 이 도구가 텍스트 편집 책임을 나눠 갖기 때문에, UI 범위 구분이 불명확하면 사용자가 혼란스럽다.
7. CSS에 data URI가 포함될 때 자산이 무거우면 복사/붙여넣기와 저장 경험이 급격히 나빠진다.
8. README, PRD, DOM 분석, 경쟁 분석 같은 문서가 현재 구현 상태보다 뒤처지면 실제 운영 판단이 흔들린다.

## 3. 타겟 사용자

- OBS Studio로 팬더TV 방송을 진행하는 스트리머
- CSS를 직접 작성하지 않고 GUI로 채팅 스타일을 만들고 싶은 사용자
- 팬더TV 기본 위젯을 쓰되, 테마 잔상 없이 깔끔한 오버레이를 원하는 사용자

## 4. 목표

### 제품 목표

1. 실제 팬더TV 위젯 DOM에 근거한 CSS를 생성한다.
2. 테마별 차이를 흡수하는 호환성 레이어를 제공한다.
3. `ul.{theme} > li...`와 `ul > li.{theme}...` 두 구조 모두에서 동작하는 CSS를 생성한다.
4. 미리보기와 실제 결과의 차이를 줄인다.
5. 스트리머가 가장 자주 만지는 오버레이 전용 옵션을 앞에 배치한 설정기 정보 구조를 제공한다.
6. 프리뷰 직접 편집은 작은 아바타/아이콘도 쉽게 선택할 수 있어야 한다.
7. 생성 CSS 뒤에 예외 보정용 커스텀 CSS를 안전하게 병합할 수 있어야 한다.
8. 아이콘/data URI를 포함하더라도 대표 프리셋 기준 CSS가 실사용 가능한 크기에 머물도록 유지한다.
9. 문서와 실제 구현 상태를 항상 일치시킨다.

### 비목표

- 백엔드, 계정, 서버 저장 기능
- 팬더TV 위젯에 JavaScript를 상시 주입하는 방식
- 다른 플랫폼(치지직, SOOP, Twitch) 동시 지원
- 실시간 원격 DOM 파싱이나 자동 패치 시스템

## 5. 제약 조건

1. 최종 산출물은 OBS 사용자 지정 CSS에 붙여넣는 CSS 문자열이어야 한다.
2. 팬더TV 위젯은 외부 Next.js 앱이며 구조 변경 가능성이 있다.
3. 인라인 스타일과 원본 테마 CSS를 덮기 위해 `!important` 사용이 필요하다.
4. OBS 브라우저 소스는 load-end 시점에 Custom CSS를 `<style>` 태그로 주입하는 모델을 가진다.
5. 사용자는 브라우저에서 단일 HTML 파일을 열어 설정기를 사용할 수 있어야 한다.
6. OBS 브라우저 환경에서 모든 시스템 폰트/외부 폰트/필터가 항상 동일하게 동작한다고 가정할 수 없다.
7. 로컬 서버의 CORS 헤더는 검증 편의 기능일 뿐, 제품 백엔드 범위가 아니다.

## 6. 핵심 해결 방식

### A. 호환성 중심 CSS 생성

- 공통 스타일만 생성하지 않고 `default`, `kakaotalk`, `neon`, `box`, `roundbox`, `balloon`, `board`를 대상으로 내부 구조 차이를 반영한다.
- 바깥 `li`뿐 아니라 `.message__box`, `.message__text`, `.notice__text`, `.haert__image`, `.heart__text` 수준까지 덮는다.
- 동일 DOM 구조를 단정하지 않고 `ul.{theme} > li...`와 `ul > li.{theme}...` 두 형태를 함께 지원한다.

### B. DOM 정합성 높은 미리보기

- `message__wrapper`, `heart__wrapper`, `chat__notice--list` 같은 실제 위젯 클래스 기반 구조를 미리보기에 사용한다.
- 일반 채팅, 후원, 알림을 별도 구조로 표현한다.
- 테마별 잔상과 내부 배경 충돌을 미리보기에서 점검할 수 있어야 한다.

### C. 실용적인 설정기 정보 구조

- `호환성`을 최상단에 배치한다.
- 팬더HP에서 직접 조정 가능한 텍스트 색상/폰트/굵기/그림자는 기본 화면에서 숨긴다.
- 대신 말풍선, 레이어, 캡슐, 아이콘, 아바타, 후원/알림 같은 오버레이 전용 옵션을 우선 노출한다.

### D. 탈출구가 있는 출력

- 생성 CSS만으로 부족한 예외 상황에 대응할 수 있도록 `추가 커스텀 CSS`를 최종 출력 뒤에 그대로 병합한다.
- 이 커스텀 CSS는 저장/불러오기 경로와 함께 움직여야 한다.

### E. 경량 자산 우선

- 대표 프리셋에서 data URI가 CSS를 과도하게 키우지 않도록, 기본 아이콘 자산은 경량 리소스를 우선 사용한다.

## 7. 기능 요구사항

### FR-01. 호환성 섹션

설정기 상단에 다음 옵션을 제공해야 한다.

- 타겟 테마: `all`, `default`, `kakaotalk`, `neon`, `box`, `roundbox`, `balloon`, `board`
- 호환성 모드: `안정형`, `균형형`, `원형 유지`
- 내부 말풍선 강제 리셋 여부
- 원본 아이콘 유지/숨김
- 원본 테마 애니메이션 제거 여부

### FR-02. DOM 정합성 미리보기

- 미리보기는 실제 위젯 구조와 유사한 클래스명을 사용해야 한다.
- 최소 3가지 메시지 유형을 포함해야 한다.
  - 일반 메시지: `.message__wrapper`
  - 후원 메시지: `.heart__wrapper`
  - 알림 메시지: `.chat__notice--list`
- hover/focus 편집 진입과 실제 CSS 생성이 같은 구조적 가정을 공유해야 한다.

### FR-03. CSS 생성 엔진

- CSS 생성 로직은 공통 규칙과 테마별 규칙으로 분리해야 한다.
- 다음 셀렉터 계층을 지원해야 한다.
  - 컨테이너: `ul.{theme}`
  - 일반 메시지: `ul.{theme} > li.message__wrapper`와 `ul > li.{theme}.message__wrapper`
  - 후원 메시지: `ul.{theme} > li.heart__wrapper`와 `ul > li.{theme}.heart__wrapper`
  - 알림 메시지: `ul.{theme} > li.chat__notice--list`와 `ul > li.{theme}.chat__notice--list`
  - 내부 요소: `.message__box`, `.message__nick`, `.message__name`, `.message__text`, `.haert__image`, `.heart__text`, `.notice__text`
  - 페이드 제어: `.hide__opacity`
- 위험도가 높은 범용 선택자는 제거하거나 대체해야 한다.

### FR-04. 스타일 값 유지와 UI 범위 분리

- CSS 생성 엔진과 프리셋은 닉네임/본문 색상, 폰트 크기, 굵기, 그림자 값을 내부적으로 유지해야 한다.
- 단일 HTML 설정기 UI는 팬더HP에서 직접 편집 가능한 텍스트 옵션을 기본 화면에서 숨길 수 있다.
- 로컬 설정기와 프리뷰 편집기는 말풍선, 캡슐, 아바타, 아이콘, 레이어, 후원/알림 같은 오버레이 전용 조정을 우선 노출해야 한다.

### FR-05. 프리뷰 직접 편집 UX

- 미리보기에서 메시지 hover/focus 시 빠른 편집 도크를 제공해야 한다.
- 같은 메시지 안에서 `말풍선`, `닉네임`, `본문`, `아바타/아이콘`을 큰 버튼으로 전환할 수 있어야 한다.
- pseudo-element 기반 아바타/아이콘도 눈에 보이는 클릭 타깃으로 편집 진입점이 제공되어야 한다.
- 플로팅 편집기는 현재 타깃 요소가 무엇인지 명확히 보여줘야 한다.

### FR-06. 로컬 설정 저장 / 불러오기

- 브라우저 localStorage 기반 저장 / 불러오기를 제공해야 한다.
- JSON 파일 내보내기 / 가져오기를 제공해야 한다.
- 저장 상태는 화면에 명확히 피드백되어야 한다.
- 계정, 서버, 원격 동기화는 제공하지 않는다.

### FR-07. 단계형 대시보드 UX

- 상단은 `STEP 1 레이아웃`, `STEP 2 프리셋`, `STEP 3 오버레이 조정` 3단계 흐름으로 제공해야 한다.
- 현재 레이아웃, 현재 프리셋, 현재 호환성/저장 상태를 즉시 읽을 수 있는 요약이 있어야 한다.
- 이전 단계 / 다음 단계 또는 최종 `CSS 복사` CTA를 같은 위치에서 제공해야 한다.
- phase 전환 로직은 취약한 문자열 매칭이 아니라 데이터 기반 선택 상태를 사용해야 한다.

### FR-08. 추가 커스텀 CSS

- 사용자가 입력한 커스텀 CSS를 최종 출력 하단에 그대로 병합해야 한다.
- 이 블록은 저장/복구와 파일 저장/불러오기 경로에 포함되어야 한다.
- 비우기 액션과 상태 피드백을 제공해야 한다.

### FR-09. 대표 프리셋 실위젯 검증

- 대표 프리셋 하나 이상을 실제 위젯 페이지에 재주입해 후원/알림 가시성과 아이콘 로드를 점검해야 한다.
- 검증 메모는 문서에 남겨 이후 회귀 판단 기준으로 사용해야 한다.

### FR-10. 출력 크기 실용성

- 기본 또는 대표 프리셋 기준 생성 CSS가 수백 KB를 불필요하게 넘지 않도록 아이콘/data URI를 경량화해야 한다.
- data URI가 포함돼도 OBS 사용자 지정 CSS에 붙여넣을 수 있을 정도의 실용적 크기를 유지해야 한다.

## 8. UX 요구사항

설정기 메뉴는 다음 순서를 기본으로 한다.

1. 호환성
2. 말풍선
3. 아이콘 / 캡슐
4. 레이아웃 확장
5. 후원 / 알림
6. 가시성 / 모션

프리셋은 남기되, 사용자 흐름의 출발점은 "팬더HP에서 텍스트 기본 설정 → 이 도구에서 호환성/오버레이 조정"이 되어야 한다.

## 9. 성공 기준

### 정성 기준

- 사용자가 팬더TV 위젯 테마를 선택하고 CSS를 붙여넣었을 때 결과가 미리보기와 크게 다르지 않다.
- `kakaotalk`, `box`, `roundbox`, `balloon`, `neon`에서 원본 내부 말풍선 잔상이 최소화된다.
- 후원/알림이 생성 CSS 때문에 의도치 않게 숨김되지 않는다.
- 아바타/아이콘 직접 편집이 정밀 클릭 없이 도달 가능하다.
- 문서가 실제 구현 상태를 과장하지 않는다.

### 정량 기준

- 위험한 범용 숨김 셀렉터 0건
- 미리보기에서 실제 클래스명 기반 메시지 유형 3종 이상 사용
- 공통 규칙과 테마별 규칙이 논리적으로 분리된 생성 구조 확보
- 브라우저 localStorage 기반 저장/복구와 JSON 파일 저장/복구가 모두 동작
- 대표 프리셋 실위젯 검증 기록 1건 이상 유지

## 11. 현재 우선순위

1. 문서와 실제 상태를 맞춘다.
2. 미리보기를 실제 DOM에 맞춘다.
3. CSS 생성 엔진을 테마별 호환성 중심으로 계속 다듬는다.
4. 대표 프리셋 실위젯 검증 메모를 누적해 회귀 기준을 만든다.

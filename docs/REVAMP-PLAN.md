# PandaTV Chat Overlay Revamp Plan

> 기준일: 2026-03-13  
> 상태: Phase 5 진행 중

## 목표

현재 프로젝트를 "공통 스타일 생성기"에서 "팬더TV 채팅 위젯 호환성 중심 생성기"로 개편한다.

## Phase 1. 문서 정합성 확보

### 목적

- 요구사항, 근거, 구현 전략을 최신 상태로 고정한다.

### 산출물

- `docs/PRD.md`
- `docs/DOM-ANALYSIS.md`
- `docs/ADR.md`
- `docs/COMPETITIVE-ANALYSIS.md`
- `README.md`

### 완료 기준

- 과장된 표현 제거
- 현재 구현 한계와 다음 단계가 문서에 명시됨

### 진행 메모

- README, PRD, DOM, ADR, 감사 문서, 경쟁 분석 문서를 현재 코드 상태와 맞춤
- 구형 한글 파일명 설명을 현재 파일명(`overlay.css`, `overlay-settings.html`) 기준으로 정리
- 대표 프리셋 재검증 메모를 문서에 반영

## Phase 2. 미리보기 DOM 현실화

### 목적

- 설정기 미리보기와 실제 팬더TV 위젯 사이의 차이를 줄인다.

### 작업

1. `preview-msg` 단순 구조 제거
2. `message__wrapper`, `heart__wrapper`, `chat__notice--list` 기반 샘플 DOM 도입
3. 테마별 내부 구조 차이를 반영할 수 있는 미리보기 클래스 설계

### 완료 기준

- 일반/후원/알림이 실제 클래스명으로 렌더링됨
- 2줄 레이아웃, 아이콘, 후원 이미지, 알림 스타일을 실제 구조에 맞춰 검증 가능

### 진행 메모

- `preview-msg` 단순 구조 제거 완료
- `message__wrapper`, `heart__wrapper`, `chat__notice--list` 기반 샘플 DOM 반영 완료
- 호환성 모드에 따라 내부 셸을 미리보기에서 부분 반영하는 1차 로직 반영 완료

## Phase 3. CSS 생성 엔진 개편

### 목적

- 테마별 내부 구조 리셋과 사용자 스타일 적용을 분리한다.

### 작업

1. 공통 규칙과 테마별 규칙 분리
2. `kakaotalk`, `box`, `roundbox`, `balloon`, `neon` 내부 요소 리셋 추가
3. 위험 선택자 제거
4. 타겟 테마 / 호환성 모드 도입
5. `ul-theme` / `li-theme` 이중 셀렉터 지원

### 완료 기준

- CSS 생성 결과가 `테마별 호환 레이어 + 사용자 스타일` 구조를 가짐
- 광범위 유틸 셀렉터에 의존하지 않음

### 진행 메모

- 타겟 테마 / 호환성 모드 / 내부 셸 리셋 / 테마 효과 리셋 옵션 연결 완료
- 정적 CSS 기본 배포본을 안정형 호환성 기준으로 교체 완료
- `ul.{theme} > li...`와 `ul > li.{theme}...`를 함께 지원하도록 보강 완료
- 대표 `👑 공주(princess)` 프리셋을 실제 위젯 페이지에 재주입해 후원/알림 가시성 재검증 완료
- 잔여 작업은 테마별 세부 리셋 정밀화와 회귀 검증 자동화

## Phase 4. 설정기 정보 구조 개편

### 목적

- 스트리머가 가장 중요한 옵션부터 조작하게 만든다.

### 새 메뉴 구조

1. 호환성
2. 말풍선
3. 아이콘 / 캡슐
4. 레이아웃 확장
5. 후원 / 알림
6. 가시성 / 모션

### 완료 기준

- 첫 조작 흐름이 "테마 선택 → 호환성 모드 → 스타일 세부 조정"으로 정리됨

### 진행 메모

- "테마 설정" → "호환성" 이름 변경 완료
- "텍스트 확장" + "레이아웃" → "레이아웃 & 커스텀 레이어" 병합 완료
- "닉네임 꾸미기" → "아이콘 & 캡슐" 이름 변경 + 말풍선 바로 뒤로 순서 변경
- "고급" → "가시성 & 모션" 이름 변경 완료
- 퀵 내비 6개 버튼으로 정리 완료
- 팬더HP와 중복되는 텍스트 계열은 standalone UI에서 기본 숨김 처리
- 3단계 phase UI를 가이드형 대시보드로 재구성 완료

## 위험 요소

- 팬더TV 위젯 DOM이 외부 서비스 변경으로 바뀔 수 있음
- OBS 브라우저 환경에서 폰트/필터 지원 차이가 날 수 있음
- 단일 HTML 파일 구조라 기능 추가 시 로직 분리 없이 비대해질 수 있음
- 실위젯 검증이 수동 중심이라 회귀 발견 속도가 느릴 수 있음

## 작업 원칙

1. 문서와 코드의 주장 수준을 맞춘다.
2. 라이브 위젯 근거 없는 확정 표현을 피한다.
3. 새 기능보다 호환성 정확도를 우선한다.
4. 미리보기와 생성 CSS는 같은 구조적 가정을 공유해야 한다.
5. 실위젯 재검증 결과는 문서로 남긴다.
